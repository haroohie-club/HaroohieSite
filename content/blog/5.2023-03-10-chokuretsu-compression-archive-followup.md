## The Compression Routine
I had a number of questions about how I actually implemented the compression routine, so I thought Iâ€™d delve into that a bit here.

I think the core process is actually pretty easy to understand: essentially, weâ€™re just reversing what the decompression routine does. For example, when decompressing a file, we might first encounter a byte with the top bit cleared and the second bit set (i.e. `0b01xxxxxx`), which according to the algorithm we reverse-engineered means that we take the lower 6 bits and add 4, then repeat the following byte that number of times (e.g., if we encounter `83 03` in the compressed buffer, we would write seven `03` bytes to the decompressed buffer). So, when compressing, we look for four or more repeated bytes in a row â€“ if we encounter that repetition, then we encode the control byte followed by the repeated byte (e.g., if we encounter `03 03 03 03 03 03 03` in the decompressed buffer, we would write `83 03` to the compressed buffer).

Thatâ€™s basically the whole process. It gets quite complicated for the â€œsliding dictionaryâ€ thatâ€™s characteristic of [LZ compression](https://en.wikipedia.org/wiki/LZ77_and_LZ78) (which I call `lookback`s in my code). For those, I essentially keep a running dictionary of every four byte sequence in the file and check to see if the current four byte sequence is in that dictionary. If it is, I insert the control code for the lookback sequence into the compressed buffer.
